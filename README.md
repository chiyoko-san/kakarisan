# ぶんたんさん（仮称）β

係決め・当番・持ち寄り・面談時間枠の「枠に名前を書く」を、登録不要のURLひとつで済ませるサインアップシート。
「調整さんの係決め版」を目指すMVPです。

- 幹事: 枠と定員を決めて作成 → **参加用URL** と **管理用URL** が発行される（アカウント不要）
- 参加者: URLを開いて、空いている枠に名前を書くだけ（アカウント不要）
- 定員に達した枠は自動で「満員」になり記入不可（同時タップの競合はトランザクションで防止）
- 名前の公開範囲: みんなに公開 / 人数のみ公開（名簿は幹事だけ）
- 締切日時・手動の受付停止・本人による取り消し・幹事による削除
- LINE共有・QRコード（紙のおたよりに貼る用）・A4印刷ビュー・CSV出力
- テンプレート: 運動会の係 / 旗振り当番 / 持ち寄り / 個人面談

## 技術スタック

Vite + React 18 / Firebase Realtime Database + 匿名認証 / GitHub Pages（ハッシュルーティング）

## セットアップ

### 1. Firebaseプロジェクトを作る

**ゲーム用（one-button）とは別の新規プロジェクト**を作ってください。保護者名などの個人情報を扱うため、ルールとデータを完全に分離します。

1. [Firebaseコンソール](https://console.firebase.google.com/) → プロジェクトを追加
2. **Realtime Database** を作成（リージョンは asia-southeast1 でOK、**ロックモード**で開始）
3. **Authentication → ログイン方法 → 匿名** を有効化
4. Realtime Database → ルール に `database.rules.json` の中身を貼り付けて公開
5. プロジェクトの設定 → 全般 → マイアプリ → Webアプリを追加し、SDK構成の値を控える

### 2. ローカル起動

```bash
cp .env.example .env   # Firebaseの値を記入
npm install
npm run dev
```

### 3. GitHub Pagesへデプロイ

```bash
npm run deploy   # dist を gh-pages ブランチへpush
```

リポジトリの Settings → Pages → Branch を `gh-pages` に設定。
`vite.config.js` の `base: './'` によりサブパス配信でもそのまま動きます。

> **将来の引っ越しメモ:** LINEで共有したときにイベント名入りのOGPカードを出すには、動的OGP（プリレンダリング）が必要です。GitHub Pagesでは不可能なので、本格運用時は Netlify / Cloudflare Pages への移行を推奨（コードはそのまま動きます）。

## データ構造（Realtime Database）

```
events/{eventId}/
  meta/            タイトル・説明・締切(epoch ms)・locked・displayMode・ownerUid・createdAt
  secret/adminKey  管理キー（読み取り不可。作成時に1回だけ書ける）
  adminProof/{uid} 「正しい管理キーを知っているuid」の登録簿（下記参照）
  slots/{slotId}   枠名・日時ラベル・定員・備考・並び順
  counts/{slotId}  記入数（数値。トランザクションで±1）
  entries/{slotId}/{entryId}  名前・メモ・uid・記入日時
```

## セキュリティ設計のポイント

**URLケイパビリティ方式。** eventId（14文字）とadminKey（26文字）は推測不能な乱数で、URLを知っていること＝権限です。この方式の性質上、URLの取り扱いがすべてなので、管理用URLは幹事以外に共有しないでください。

**定員の強制は二重。** クライアントは `counts/{slotId}` への `runTransaction` で「読んで・確認して・+1」を原子的に行い、同時タップでも定員超過しません。さらにルール側でも `counts <= capacity` と「一般ユーザーは±1のみ」「締切後・ロック中は増加不可」を強制しているため、細工したクライアントでも突破できません。

**「自分の記入だけ取り消せる」は匿名認証で強制。** 記入には端末ごとの匿名uidが刻まれ、ルールが `entry.uid === auth.uid` の削除のみ許可します（幹事は除く）。

**幹事の証明は adminProof 方式。** ルールは「`adminProof/{自分のuid}` に、`secret/adminKey` と一致する値しか書けない」ようになっています。つまり書き込みに成功したuid＝正しい管理URLの持ち主。以降の管理操作（枠編集・記入削除・メタ変更）はこの登録簿の存在チェックで判定されます。サーバーコードなしで「秘密の照合」を実現するための仕掛けです。

## 既知の割り切り（v2候補）

- **「人数のみ公開」モードは表示上の配慮です。** entriesはURLを知る人なら技術的には読めるため、悪意ある技術者には名前が見えます。PTAの実際の脅威モデル（人の目・気まずさ）には十分ですが、本気の秘匿が必要になったらCloud Functions化が必要です。
- **幽霊枠。** 記入処理が途中で切断されると、人数表示と実記入がずれることが稀にあります。管理ページの「人数を再集計」で修復できます。
- **取り消しは同一端末・ブラウザのみ。** 匿名認証のuidが端末に紐づくため。間違えた場合は幹事が削除します。
- **自動削除は未実装。** 役目を終えた募集は手動で枠を削除してください（v2で「締切後90日で自動削除」を予定）。
- WebのAPIキーは公開前提の識別子です（秘密ではない）。防御線はセキュリティルールです。
